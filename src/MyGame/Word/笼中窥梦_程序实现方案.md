# 《笼中窥梦》类游戏程序实现方案

## 1. 文档目标

本文面向程序、技术策划、TA 和原型阶段的主程，目标是回答：

- 《笼中窥梦》这类“视角拼接解谜”从程序上怎么拆
- 哪些效果适合真做，哪些更适合伪造
- Unity 下做原型时，最小可行方案应该是什么
- 量产时会遇到哪些系统和工具问题

本文重点是“做类似体验的原创技术路线”，不是对原作的逆向工程。

## 2. 技术目标先定义清楚

这类游戏从玩家视角看起来很神奇，但从实现角度要尽量拆成可控的“作者驱动系统”。

前期技术目标不应该是：

- 做一个完全自由、自动推导一切拼接关系的系统

而应该是：

- 做一个可编辑、可验证、可调试的视角拼接框架

换句话说，优先级应当是：

1. 谜题可 authoring
2. 反馈可控制
3. 玩家行为可验证
4. 美术和关卡能迭代
5. 性能最后再系统优化

## 3. 先给结论：推荐“半自动、强配置、重作者标注”的方案

如果你真的要做一款同类项目，我不建议一开始就追求“纯几何自动判定”。

推荐路线是：

- 视觉呈现上使用真实 3D 场景
- 谜题判定上使用作者预设的锚点/窗口/条件
- 交互推进上使用状态机和事件链
- 调试工具上提供投影预览、对位阈值可视化、谜题链路回放

这是最稳的原因：

- 这类谜题本来就是人工设计出来的
- 自动判定越强，调试成本越高
- 真正影响体验的不是“数学上是否完美”，而是“作者能否稳定地做出 Wow Moment”

## 4. 系统分层建议

建议拆成 6 层：

### 1. 世界容器层

负责：

- 立方体或其他容器的旋转
- 面切换
- 当前可见场景管理
- 镜头约束和过渡

### 2. 场景表现层

负责：

- 每个面的 3D 场景摆放
- 关键物体显隐
- 光照、材质、粒子、动画状态
- 场景内部可交互点

### 3. 视角拼接判定层

负责：

- 判断两个物体是否在当前视角下“足够对齐”
- 判断是否满足额外条件
- 输出对齐程度、成功/失败、候选提示

### 4. 谜题状态层

负责：

- 谜题是否激活
- 已完成步骤
- 连锁步骤条件
- 是否解锁新场景/新交互

### 5. 叙事与反馈层

负责：

- 成功对位后的演出
- 收集物解锁
- 照片/回忆/文字片段的管理
- 提示系统

### 6. 工具与调试层

负责：

- 编辑器中的锚点摆放
- 角度窗口配置
- 可视化投影重叠结果
- 谜题状态重置和跳转

## 5. Unity 中的三种实现路线

## 5.1 路线 A：纯作者驱动的“假拼接”

做法：

- 所有谜题都是预定义的
- 每个谜题给一组或多组目标角度区间
- 玩家旋转到目标角度附近时，直接触发可拼接状态
- 成功后播放演出和状态推进

优点：

- 成本最低
- 最易调试
- 最适合验证玩法

缺点：

- 缺乏一般化能力
- 新增谜题大量依赖手工配置
- 复杂错位和动态物体会比较难扩展

适用阶段：

- 原型阶段
- 第一版 vertical slice

## 5.2 路线 B：投影空间判定

做法：

- 每个可拼接物体都设置一个或多个 `PuzzleAnchor`
- 运行时将锚点投影到屏幕空间
- 比较两组锚点的 2D 距离、角度差、尺寸比、遮挡关系
- 达到阈值时判定为可连接

优点：

- 更符合玩家直觉
- 不依赖固定角度，可以支持多个解法窗口
- 更适合做“接近正确时高亮提示”

缺点：

- 需要更细的阈值管理
- 对透视相机和物体尺度比较敏感
- 调试成本高于路线 A

适用阶段：

- 进入正式开发后
- 需要更多谜题复用能力时

## 5.3 路线 C：几何/图像层自动识别

做法：

- 用轮廓匹配、遮罩重叠、甚至图像识别去自动判断两个物体是否视觉一致

不推荐原因：

- 开发成本高
- 不稳定
- 很难解释为什么某次成功某次失败
- 对作者和 QA 极不友好

结论：

除非项目本身就在做研究型图形技术，否则不建议把主线玩法建立在这一层。

## 6. 推荐的 MVP 技术方案

推荐采用 `路线 A + 路线 B 的混合方案`：

- 第一阶段使用角度窗口快速搭出谜题
- 第二阶段为关键拼接补充屏幕空间锚点判定
- 最终成功条件采用“作者配置 + 自动计算结果”双重约束

也就是：

- 自动计算负责给出“现在很接近了”
- 作者配置负责决定“这里是否真的允许成功”

这样可以同时得到：

- 更自然的交互手感
- 更稳定的内容生产

## 7. 关键数据结构设计

下面给出推荐的数据模型。

## 7.1 PuzzleAnchor

用于表示物体上参与拼接的关键点或关键方向。

建议字段：

```csharp
public class PuzzleAnchor : MonoBehaviour
{
    public string anchorId;
    public string groupId;
    public AnchorKind kind;
    public Transform point;
    public Vector3 forward;
    public Vector2 expectedScreenSizeRange;
}
```

可扩展用途：

- 点对点对位
- 方向一致性验证
- 轮廓近似验证
- 提示系统高亮

## 7.2 PuzzleLinkConfig

用于描述一条“允许发生的拼接关系”。

```csharp
[Serializable]
public class PuzzleLinkConfig
{
    public string linkId;
    public string fromAnchorId;
    public string toAnchorId;
    public float maxScreenDistance;
    public float maxAngleDelta;
    public Vector2 validCameraYawRange;
    public Vector2 validCameraPitchRange;
    public string[] requiredFlags;
    public string successEventId;
}
```

说明：

- `requiredFlags` 用于限制时序，避免玩家跳题
- `successEventId` 进入统一事件链

## 7.3 PuzzleStep / PuzzleChain

用于管理谜题多步推进。

```csharp
[Serializable]
public class PuzzleStep
{
    public string stepId;
    public string[] availableLinks;
    public string[] requiredStateFlags;
    public string[] grantStateFlags;
    public string onStepStartEvent;
    public string onStepCompleteEvent;
}
```

一个章节通常由多个 `PuzzleStep` 串成 `PuzzleChain`。

## 7.4 StoryCollectible

用于管理照片、回忆、录音等补叙事元素。

```csharp
[Serializable]
public class StoryCollectible
{
    public string collectibleId;
    public string unlockCondition;
    public bool affectEnding;
    public string previewAsset;
    public string fullContentAsset;
}
```

## 8. 核心判定逻辑建议

## 8.1 屏幕空间锚点判定

最实用的方案是“把 3D 难题降维到 2D 屏幕空间”。

基本流程：

1. 取当前主相机
2. 将两个锚点投影到屏幕坐标
3. 比较它们的像素距离
4. 比较锚点朝向投影后的方向差
5. 检查可见性和遮挡
6. 若满足阈值，则进入“可连接”状态

伪代码：

```csharp
bool IsLinkMatched(Camera cam, PuzzleAnchor a, PuzzleAnchor b, PuzzleLinkConfig config)
{
    Vector3 sa = cam.WorldToScreenPoint(a.point.position);
    Vector3 sb = cam.WorldToScreenPoint(b.point.position);
    if (sa.z <= 0 || sb.z <= 0)
        return false;

    float screenDistance = Vector2.Distance(sa, sb);
    if (screenDistance > config.maxScreenDistance)
        return false;

    Vector3 dirA = cam.WorldToScreenPoint(a.point.position + a.point.forward) - sa;
    Vector3 dirB = cam.WorldToScreenPoint(b.point.position + b.point.forward) - sb;
    float angleDelta = Vector2.Angle(dirA, dirB);
    if (angleDelta > config.maxAngleDelta)
        return false;

    return true;
}
```

## 8.2 为什么不要只用中心点距离

只比两个点的距离会有很多误判：

- 轮廓看起来没对上，但点碰巧很近
- 大小完全不一致
- 方向相反
- 被遮挡的物体也会误触发

因此至少还要引入：

- 方向差
- 尺寸范围
- 可见性
- 作者配置的角度窗口

## 8.3 成功触发要分两段

建议不要一满足条件就直接成功，而是拆成：

- `NearMatch`：接近正确，给轻提示
- `PerfectMatch`：满足全部条件，允许吸附/确认/触发

这样好处很大：

- 玩家更容易理解自己接近正确答案
- 可以做轻微吸附，减少精确旋转挫败感
- 演出节奏更可控

## 9. 容器与镜头实现建议

## 9.1 世界容器怎么做

最直接方案有两种：

### 方案 1：真实立方体 + 内部场景

- 在容器内放置多个面向不同方向的场景
- 主相机围绕容器观察
- 每个面本质上是一个空间窗口

优点：

- 概念直观
- 易于理解旋转关系

缺点：

- 场景布局和遮挡管理会更复杂

### 方案 2：容器只是 UI 包装，内部是独立场景组

- 玩家看到的是一个立方体表现
- 实际上每个面对应单独场景节点与虚拟相机
- 通过裁切、遮罩、显示规则伪装成同一个立方体内部

优点：

- 更容易控制表现和优化
- 每个面可以独立迭代

缺点：

- 需要更细的镜头与空间统一规范

推荐：

如果项目早期以可控性为主，优先用 `方案 2`。

## 9.2 相机建议

公开资料中开发团队提到他们最终采用透视相机而非正交相机，因为透视更能体现深度，但也带来了尺寸一致性难题。

如果你要做类似体验：

- 原型阶段建议先用透视相机
- FOV 尽量固定，不要频繁变化
- 把“哪些谜题依赖透视效果”提前定义清楚

原因：

- 透视是这类体验“神奇感”的重要来源
- 但透视也会导致作者在摆放物体时非常痛苦

所以需要配套编辑器工具来可视化结果。

## 10. 必做编辑器工具

这类项目是否能做下去，很大程度取决于工具，不取决于底层数学是否优雅。

至少要有以下工具：

### 1. 锚点投影预览

在 SceneView 中显示：

- 锚点位置
- 当前相机投影点
- 两锚点屏幕距离

### 2. 有效角度区间预览

在编辑器里直接看到：

- 哪些 yaw/pitch 范围内允许判定
- 当前旋转是否落在有效窗口中

### 3. 拼接候选列表

当前视角下显示：

- 哪些 link 正在接近成功
- 分数是多少
- 差在哪个条件

### 4. 谜题链路调试器

支持：

- 重置某一步
- 跳过某一步
- 强制设置 flag
- 回放成功演出

### 5. 提示系统预演

能在编辑器里预览：

- 第 1 层提示会高亮什么
- 第 2 层提示会指向什么
- 第 3 层提示会显示什么文本或动画

## 11. 谜题状态管理

不要把谜题状态散落在各种 MonoBehaviour 上。  
建议集中到一个 `PuzzleRuntimeState` 或章节状态管理器中。

应当至少记录：

- 当前章节
- 当前步骤
- 已解锁 flag
- 已完成 link
- 已拾取收集物
- 当前提示等级
- 最近尝试失败次数

这样可以支撑：

- 提示动态解锁
- 自动存档
- QA 快速复现
- 分析玩家卡点

## 12. 提示系统程序实现建议

提示系统建议做成独立服务，不要写死在谜题脚本里。

推荐三层结构：

### 层 1：关注引导

功能：

- 高亮可疑区域
- 减少搜索空间

### 层 2：关系提示

功能：

- 暗示某两个元素存在联系
- 但不直接说明操作方式

### 层 3：操作意图提示

功能：

- 告诉玩家“你想让某物接到某物上”
- 仍尽量不直接给最终角度

程序触发条件可基于：

- 停留时长
- 旋转次数
- 重复失败次数
- 是否长时间没有触发新状态

## 13. 量产期会遇到的主要技术问题

### 1. 透视下尺寸不一致

同样两个形状，在不同深度和角度下看起来完全不一样。  
解决方式：

- 关键拼接对象允许局部非真实比例
- 通过作者工具从目标视角反推物体摆位
- 必要时为谜题专门做“只为该视角成立”的模型

### 2. 物体遮挡关系复杂

玩家看到的是“屏幕上的轮廓”，不是世界坐标。  
解决方式：

- 增加可见性检测
- 对关键对象设置独立渲染层
- 必要时在关键角度做视觉让步

### 3. 修改一题牵动全局

因为谜题常常是链式的。  
解决方式：

- 每章限制依赖深度
- 谜题链配置化
- 为每章建立依赖图

### 4. 测试成本高

玩家卡关不一定是 bug，也可能是设计不可理解。  
解决方式：

- 记录玩家视角轨迹
- 记录接近成功但未成功的 link
- 记录使用提示的时机

### 5. 性能问题

如果每个面都是真实场景并且动态渲染，成本会很高。  
解决方式：

- 优先只更新当前相关面
- 非关键面降频更新或静态化
- 关键动画才开启实时刷新
- 充分使用烘焙、LOD、剔除

## 14. 更适合团队落地的制作流程

推荐流程如下：

### 阶段 1：玩法原型

目标：

- 只做 3 个场景
- 只做 5 个链接
- 不做完整叙事
- 验证玩家能否理解规则

技术重点：

- 容器旋转
- 角度窗口
- NearMatch / PerfectMatch

### 阶段 2：Vertical Slice

目标：

- 做 10 到 15 分钟可玩内容
- 加入完整提示系统
- 加入一条简单叙事线

技术重点：

- 屏幕空间锚点
- 章节状态
- 收集物系统
- 调试工具

### 阶段 3：量产工具化

目标：

- 让关卡和美术能独立生产内容

技术重点：

- Editor 工具
- 配置导入导出
- Link 批量校验
- 自动化回归测试

## 15. 一个推荐的最小原型清单

如果你现在就要在 Unity 里开工，第一周建议只做下面这些：

- 可旋转容器
- 3 个场景面
- 每个场景 2 个可交互物体
- 5 条 `PuzzleLinkConfig`
- 1 个链式两步谜题
- 1 套接近成功时的轻提示
- 1 套成功后的吸附和演出
- 1 个简单的谜题状态查看面板

只要这套东西跑通，就足以判断项目有没有继续扩张的价值。

## 16. 和《笼中窥梦》相关的关键技术启发

从公开采访里，可以提炼出几个非常重要的实现启发：

- 灵感层面来自 `stencil buffer` 的立方体表现，但真正产品化依赖的是完整关卡设计
- 他们采用透视相机而非正交相机，以增强深度感
- 由于透视导致尺寸一致性困难，他们为摆放和缩放制作了自定义工具
- 谜题设计高度依赖人工反复试验，而不是自动生成
- 提示系统是逐步迭代出来的，不是最后随手补上的

这意味着：

你的技术方案不能只关注“渲染怎么做”，更要关注“作者怎么做题”。

## 17. 对 Unity 项目的最终技术建议

如果团队规模不大，建议坚持下面的技术原则：

- `作者优先`：判定宁可保守，也要可控
- `工具优先`：先做编辑器工具，再扩谜题数量
- `状态优先`：所有谜题推进必须可回放、可重置、可追踪
- `提示优先`：把提示当主系统，不是辅助功能
- `混合判定`：角度窗口 + 屏幕空间锚点，不要迷信全自动

## 18. 最终结论

《笼中窥梦》类游戏从程序角度看，并不是一个“高深图形算法项目”，而是一个：

- 需要适度图形技巧
- 更依赖内容 authoring
- 极度依赖调试工具
- 极度依赖谜题可理解性验证

最适合落地的方案，不是追求技术炫技，而是建立一套能让设计师、美术、程序共同生产“视角惊喜”的内容管线。

一句话总结：

程序的职责不是自动生成神奇，而是把设计师想要的神奇稳定、可控、可调试地交付出来。
